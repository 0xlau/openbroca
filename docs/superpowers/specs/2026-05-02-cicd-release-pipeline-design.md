# CI/CD Release Pipeline Design

**Date:** 2026-05-02
**Status:** Approved (pending user review of this spec)
**Owner:** Timothy Lau

## Goal

Set up a production-grade open-source release pipeline for the Openbroca desktop app so that:

1. Every push to a branch is validated (lint / typecheck / test / build).
2. Cutting a release is a single PR merge — no manual versioning, no manual changelog writing.
3. Releases produce signed + notarized macOS DMGs (arm64 + x64), Windows NSIS installer (x64), and Linux AppImage + deb, all uploaded to a public GitHub Release with auto-generated release notes.
4. End users on macOS/Windows get auto-update via `electron-updater` reading directly from GitHub Releases.

## Non-Goals

- **Homebrew distribution** — explicitly deferred. The CI design leaves a clean extension point (a future `homebrew-tap-sync.yml` workflow) so adding it later is additive.
- **Windows code signing** — not pursued. EV cert cost is unjustified at this stage; SmartScreen reputation will warm up organically.
- **Auto-publish on every commit** — release-please's "Release PR" model preserves a human gate before publishing.
- **Multi-package releases** — only `apps/desktop` is released; library packages stay internal.

## Architecture

```
Developer commit (Conventional Commits) → push to main
       │
       ▼
  ci.yml (push + PR)
       │  · pnpm install (cached)
       │  · turbo lint + typecheck + test + build
       │  └─ runs on ubuntu-latest
       │
       ▼
  release-please.yml (push to main)
       │  · maintains a permanent "Release PR" accumulating
       │    Conventional Commits since last release
       │  · on merge: bumps apps/desktop/package.json,
       │    writes CHANGELOG.md, creates tag vX.Y.Z,
       │    publishes a GitHub Release with notes
       │
       ▼ (release.published event)
  release.yml (matrix: macos-26 / windows-latest / ubuntu-latest)
       │  · pnpm install --frozen-lockfile
       │  · electron-builder install-app-deps (rebuild native modules)
       │  · electron-builder --publish=always
       │     → uploads dmg/exe/AppImage/deb + latest-*.yml
       │       to the GitHub Release
       │
       ▼
  GitHub Release (public, downloadable)
       │
       ▼
  Installed clients ←── electron-updater polls latest-*.yml
```

## Workflow Files

All new files under `.github/workflows/`.

### `ci.yml` — Validation on every push/PR

**Trigger:** `push` to any branch, `pull_request` to `main`.
**Runner:** `ubuntu-latest` (single platform sufficient for type/lint/test).
**Steps:**

1. Checkout
2. Setup pnpm (`pnpm/action-setup@v4`, version pinned to `10.32.1` matching `packageManager` field)
3. Setup Node.js 22 with `cache: 'pnpm'`
4. Restore turbo cache via `actions/cache@v4` (key: `turbo-${{ runner.os }}-${{ github.sha }}`, restore-keys: `turbo-${{ runner.os }}-`)
5. `pnpm install --frozen-lockfile`
6. `pnpm turbo run lint typecheck test build` — single turbo invocation lets it parallelize internally

**Concurrency:** group by `ci-${{ github.ref }}` with `cancel-in-progress: true` so new pushes to a branch supersede in-flight runs.

### `release-please.yml` — Maintain Release PR + create GitHub Release

**Trigger:** `push` to `main`.
**Runner:** `ubuntu-latest`.
**Permissions:** `contents: write`, `pull-requests: write`.
**Single step:** `googleapis/release-please-action@v4` configured to read `.github/release-please-config.json` and `.github/.release-please-manifest.json`.

### `release.yml` — Build + publish to GitHub Release

**Trigger:** `release.published` (the release release-please just created).
**Permissions:** `contents: write`.
**Strategy:** matrix across three runners.

```yaml
strategy:
  fail-fast: false
  matrix:
    include:
      - os: macos-26
        build_args: --mac --arm64 --x64
      - os: windows-latest
        build_args: --win --x64
      - os: ubuntu-latest
        build_args: --linux
```

**Steps (per runner):**

1. Checkout the tag (`ref: ${{ github.event.release.tag_name }}`)
2. Setup pnpm + Node 22 (same versions as CI)
3. Restore turbo + pnpm caches
4. `pnpm install --frozen-lockfile` — rebuilds native modules for host arch
5. `pnpm --filter desktop exec electron-builder install-app-deps` — rebuilds native modules against Electron's Node ABI
6. `pnpm --filter desktop run build` — typecheck + electron-vite build
7. `pnpm --filter desktop exec electron-builder ${{ matrix.build_args }} --publish=always`

**Environment variables for the build step:**

`GH_TOKEN` is set on every matrix entry (electron-builder needs it on all platforms to upload artifacts). The five Apple variables are scoped to the macOS matrix entry only — they're inert noise on Windows/Linux.

```yaml
- name: Build & publish
  env:
    GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    # Apple secrets only meaningful on macOS; conditionally set so we don't
    # leak them to the win/linux runners' env (defense in depth).
    CSC_LINK: ${{ matrix.os == 'macos-26' && secrets.CSC_LINK || '' }}
    CSC_KEY_PASSWORD: ${{ matrix.os == 'macos-26' && secrets.CSC_KEY_PASSWORD || '' }}
    APPLE_ID: ${{ matrix.os == 'macos-26' && secrets.APPLE_ID || '' }}
    APPLE_APP_SPECIFIC_PASSWORD: ${{ matrix.os == 'macos-26' && secrets.APPLE_APP_SPECIFIC_PASSWORD || '' }}
    APPLE_TEAM_ID: ${{ matrix.os == 'macos-26' && secrets.APPLE_TEAM_ID || '' }}
  run: pnpm --filter desktop exec electron-builder ${{ matrix.build_args }} --publish=always
```

When the Apple secrets are absent (initial setup, before certs procured), electron-builder skips signing and produces unsigned DMGs without failing the workflow. This is intentional: it lets the entire pipeline land + run end-to-end before the developer-account paperwork is done.

### `.github/.release-please-manifest.json`

```json
{ "apps/desktop": "0.1.0" }
```

**Tag format:** `v0.1.0`, `v0.2.0`, ... (no component prefix because `include-component-in-tag: false`).

**On `package-name` vs `name`:** `package-name: openbroca` is release-please's display label (used in PR titles like *"chore(main): release openbroca 0.1.0"*); `apps/desktop/package.json#name` becomes `openbroca-desktop` to disambiguate from a possible future `openbroca-cli`. The mismatch is intentional and harmless.

**On `CHANGELOG.md`:** the file doesn't exist yet. release-please's first run creates it automatically at `apps/desktop/CHANGELOG.md` when the initial Release PR is merged — no manual seeding needed.

**Commit scope behavior:** release-please only counts commits whose paths touch `apps/desktop/**` toward the version bump. Changes to `packages/**`, `docs/**`, `.github/**` won't trigger a release on their own — by design.

## Native Modules Handling

`apps/desktop` ships with native modules: `audify`, `keytar`, `uiohook-napi`, plus the workspace package `@openbroca/audio-capture`.

**Why this matters:** native modules must be compiled against (a) the host OS/arch and (b) Electron's specific Node ABI. Both happen in the release runner.

**Mechanism:**

1. `pnpm/action-setup@v4` + `pnpm install --frozen-lockfile` rebuilds native modules for the **host Node**, governed by `pnpm.onlyBuiltDependencies` in root `package.json` (already configured).
2. `pnpm --filter desktop exec electron-builder install-app-deps` then **rebuilds against Electron's Node ABI**. This is the canonical electron-builder workflow for monorepos and handles pnpm's hoisted layout correctly (whereas `electron-rebuild` does not).
3. For the macOS arm64 host building x64 binaries, electron-builder cross-compiles via Rosetta (preinstalled on `macos-26` runners). If a specific native module fails to cross-compile, fix is module-by-module (likely none, but flagged as a known risk).

## CI Cache Strategy

```yaml
- uses: pnpm/action-setup@v4
  with:
    version: 10.32.1
- uses: actions/setup-node@v4
  with:
    node-version: 22
    cache: 'pnpm'
- uses: actions/cache@v4
  with:
    path: |
      .turbo
      apps/desktop/node_modules/.cache
    key: turbo-${{ runner.os }}-${{ github.sha }}
    restore-keys: |
      turbo-${{ runner.os }}-
```

Expected timings: cold ~3 min, hot ~1 min for `ci.yml`; release.yml ~8–12 min total (macOS notarization is the long pole at 5–10 min).

## Required GitHub Secrets

| Secret | Required for | Configured when |
|---|---|---|
| `GITHUB_TOKEN` | release-please, electron-builder upload | Auto-injected, no setup |
| `CSC_LINK` | macOS signing (base64-encoded .p12) | After Apple Developer Program enrollment |
| `CSC_KEY_PASSWORD` | macOS signing | Same time as `CSC_LINK` |
| `APPLE_ID` | macOS notarization | Same time |
| `APPLE_APP_SPECIFIC_PASSWORD` | macOS notarization (generated at appleid.apple.com) | Same time |
| `APPLE_TEAM_ID` | macOS notarization | Same time |

Pipeline is functional without the five Apple secrets — produces unsigned DMGs. This is intentional to allow end-to-end pipeline validation before procuring the developer account.

## Risks / Known Issues

1. **Native module x64 cross-compilation on arm64 host** — if any of `audify` / `keytar` / `uiohook-napi` / `audio-capture` lacks proper x64 support on an arm64 host, the macOS x64 DMG build will fail. Mitigation: implementation Stage tests the workflow against a real PR before tagging a release; if cross-compile breaks, fall back to splitting macOS into two runner jobs (one per arch).
2. **Unsigned Windows installer triggers SmartScreen** — accepted trade-off; documented in README installation section.
3. **release-please needs first commit on `main` after install to seed manifest** — first run will create the initial Release PR; merging that PR creates `v0.1.0` and triggers the first end-to-end release.
4. **`@openbroca/audio-capture` workspace dep** — its build pipeline (if it has one beyond `tsc`) must be runnable in CI. Confirmed by existing `turbo build` working locally; CI uses the same task.

## Out of Scope / Future Extensions

These are explicitly **not** in this design but are easy to add later:

- **Homebrew Cask distribution** — add `homebrew-tap-sync.yml` triggered on `release.published`, opens a PR to a `homebrew-openbroca` tap repo updating the cask version + sha256.
- **Windows code signing** — add `WIN_CSC_LINK` + `WIN_CSC_KEY_PASSWORD` secrets when an EV cert is procured; electron-builder picks them up automatically.
- **Nightly/canary builds** — add a scheduled workflow that runs `release.yml`-equivalent against `main` HEAD with prerelease tag.
- **PR build previews** — add an opt-in `workflow_dispatch` job that builds and attaches DMG/EXE/AppImage to a PR comment for testing.
- **Linux signing (Snap/Flatpak)** — only worth doing if Linux usage grows.

## Implementation Stages (preview, full plan to follow)

1. **Stage 1: Identity + signing-ready config** — update `electron-builder.yml`, `dev-app-update.yml`, `apps/desktop/package.json`; add `LICENSE`; verify `pnpm --filter desktop build:mac` still produces a working DMG locally with the new `appId`/`productName`.
2. **Stage 2: ci.yml** — add validation workflow; merge a PR to confirm it goes green.
3. **Stage 3: release-please** — add config + manifest + workflow; first push opens initial Release PR.
4. **Stage 4: release.yml** — add build/publish workflow.
5. **Stage 5: Auto-updater wiring + README rewrite** — add `autoUpdater.checkForUpdatesAndNotify()` in main, rewrite README with installation section.
6. **Stage 6: First end-to-end release** — merge the Release PR, watch matrix runners, debug any native-module issues, verify users can download from the GitHub Release page.
7. **Stage 7 (deferred to when Apple cert arrives)** — add the five Apple secrets, retag a 0.1.1 patch release to validate signed/notarized output.
